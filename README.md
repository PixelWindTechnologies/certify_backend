# PixelWind Certificate Engine

A full-stack platform for **Pixel Wind Technologies** to issue, track, and
publicly verify internship certificates across multiple colleges, courses,
and batches.

- **Frontend:** Next.js 14 (App Router) + TypeScript + Tailwind CSS
- **Backend:** FastAPI + SQLAlchemy + Alembic
- **Database:** PostgreSQL
- **Certificates:** ReportLab (PDF) + `qrcode` (QR codes), generated by a
  background scheduler
- **Auth:** JWT access + refresh tokens, role-based access control

---

## 1. Quick start (Docker)

The fastest way to run the whole stack:

```bash
docker compose up --build
```

This starts PostgreSQL, the FastAPI backend (running migrations and seed
data automatically on boot), and the Next.js frontend.

- Frontend: http://localhost:3000
- Backend API: http://localhost:8000
- API docs (Swagger): http://localhost:8000/docs

### Default super admin login

A super admin account is seeded automatically on first boot:

```
Email:    admin@pixelwind.in
Password: ChangeMe123!
```

**Change this password immediately in any environment beyond local testing.**
It is controlled by `FIRST_SUPER_ADMIN_EMAIL` / `FIRST_SUPER_ADMIN_PASSWORD`
in the backend environment.

A demo college, course, and batch are also seeded so you can explore the
app immediately without manual setup.

### Student login accounts (V1.1)

Every student gets a login account automatically — created the moment
they're added (one at a time or via Excel import). There's no email
integration yet, so:

- **Username:** the student's email address
- **Default password:** `PW@` + the last 4 digits of their phone number
  (e.g. phone `9876543210` → password `PW@3210`)
- Students are required to set a real password the first time they log in.
- If a student forgets their password, an admin can reset it from
  **Students → Reset password** — this resets it back to the same default
  format and forces a change on next login. The temporary password is
  shown once to the admin to pass along; it isn't emailed anywhere.

## 2. Running locally without Docker

### Backend

```bash
cd backend
cp .env.example .env        # adjust DATABASE_URL etc. as needed
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
alembic upgrade head
python -m app.scripts.seed
uvicorn app.main:app --reload
```

Requires a running PostgreSQL instance matching `DATABASE_URL`.

### Frontend

```bash
cd frontend
cp .env.example .env.local  # set NEXT_PUBLIC_API_URL if not http://localhost:8000
npm install
npm run dev
```

## 3. Architecture overview

```
frontend/   Next.js app — three role-scoped dashboards (Super Admin,
            College Admin, Student) plus a public certificate
            verification page at /verify/[certificateId]
backend/
  app/api/routes/   REST endpoints, grouped by resource
  app/models/       SQLAlchemy ORM models
  app/schemas/       Pydantic request/response schemas
  app/services/      Business logic: certificate rendering, QR codes,
                      Excel import, internship ID generation, the
                      scheduled certificate-generation job, audit logging
  alembic/            Database migrations
``'

### Roles

| Role           | Can do |
|----------------|--------|
| `SUPER_ADMIN`  | Everything: manage colleges, courses, certificate templates, the authorized signature, revoke certificates, view audit logs |
| `COLLEGE_ADMIN`| Manage their own college's students, batches, and enrollments; download (but not revoke) certificates |
| `STUDENT`      | View their own enrollments and download their own certificates |

### Internship ID format

```
PW/VSP/<CourseCode>/<YYMM>/B<BatchNumber>/<StudentNumber>
e.g. PW/VSP/DA/2606/B01/0001
```

### Certificate generation

A background job (APScheduler) runs on an interval
(`CERTIFICATE_JOB_INTERVAL_SECONDS`, default 120s) and generates a
certificate for any enrollment where:

1. `status == COMPLETED`
2. `certificate_approval == APPROVED`
3. No certificate exists for it yet

A super admin can also trigger this manually from
**Certificates → Run generation now**. Before approving an enrollment, an
admin can use **Preview certificate** (in the Enrollments edit panel) to
see exactly what will be generated — nothing is saved or counted as
issued until the real generation step runs.

As of V1.1, certificates carry no QR code and no visible ID strings —
just branding, the student's name, course, duration, grade, and the
authorized signature. Certificate and internship IDs still exist in the
database and power verification below; they're simply not printed on the
document itself.

### Verification

Certificates are looked up **only by certificate ID** (never by student
ID) at `/verify/{certificate_id}` — both as a backend API endpoint and as
a public, no-login frontend page. The QR code on every certificate PDF
encodes this verification URL. Every lookup is logged.

### Future-ready modules

`future_attendance`, `future_assessments`, `future_projects`, and
`future_placements` tables exist in the schema (with migrations) but have
no API routes or UI yet — they're scaffolded for upcoming feature work,
per the original project brief.

## 4. Excel student import

Admins can download a blank template (**Students → Import Excel →
Download template**) and re-upload it. Required columns: Student Name,
Father Name, Phone Number, Email Address, Graduation Year, College Name,
Course Name, Batch Name. A student is matched by (college, email) so
re-importing the same student into a new course never creates a
duplicate — it just adds a new enrollment.

---

## 5. Storage backends

Certificates, QR codes, templates, and signatures are saved through a
storage abstraction (`STORAGE_BACKEND=local|s3|r2`). Local disk is the
default and requires no extra configuration; set the `S3_*` variables in
`.env` to point at S3 or Cloudflare R2 instead.
